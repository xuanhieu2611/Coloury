Status of PRD §8 M5 (the "stretch" milestone). Scope for this pass (chosen with

Batch editing and cloud storage of projects/presets are **deferred** (see below).

## 1. RAW support — ✅ COMPLETE & VERIFIED

Decode camera RAW files in the browser via **LibRaw compiled to WASM**
(`libraw-wasm`), demosaiced to an 8-bit sRGB canvas that flows through the exact
same edit/export pipeline as a JPEG. The AI/recipe/renderer are untouched.

### Files
| File | Role |
|---|---|
| `src/lib/raw.ts` | `isRaw(file)` (extension allowlist) + `decodeRaw(file)` → `{canvas, width, height, exif}`. Demosaics via the vendored client, packs LibRaw's `colors`-channel buffer into RGBA, paints a full-res canvas, and maps LibRaw metadata → our `ExifData` (`toExif`). |
| `src/lib/libraw-client.ts` | **Vendored** TS port of `libraw-wasm`'s worker RPC wrapper. Identical protocol; the *only* change is the worker is spawned from the fixed string path `/libraw/worker.js` instead of `new Worker(new URL('./worker.js', import.meta.url))`. Types come from `libraw-wasm` via a type-only import (erased at compile). |
| `scripts/copy-libraw.mjs` | Copies `worker.js`/`libraw.js`/`libraw.wasm` from `node_modules/libraw-wasm/dist` → `public/libraw/` (stripping sourcemap refs). Runs on `postinstall`/`predev`/`prebuild` so a fresh `npm ci` re-vendors the matching version. |
| `public/libraw/` | The vendored runtime (served statically). **Not** hand-edited — regenerated by the script. |
| `src/lib/store.ts` | `loadImageFile` branches on `isRaw(file)` → `decodeRaw`, builds the downscaled preview, returns a `LoadedImage`. `LoadedImage.element` widened to `HTMLImageElement | HTMLCanvasElement`. |
| `src/components/Uploader.tsx` | Accepts RAW extensions in the file picker + "RAW" in the copy. |

### ⚠️ The build-hang gotcha (important — cost real time)
`libraw-wasm`'s npm entry constructs its worker with
`new Worker(new URL('./worker.js', import.meta.url), {type:'module'})`. In dev
this is fine, but **`next build` (Turbopack) DEADLOCKS** trying to trace+bundle
that worker and its 1.4 MB wasm — the build sits at "Creating an optimized
production build …" at **0% CPU forever**. Confirmed by bisection: stubbing the
`import('libraw-wasm')` drops the build to ~950 ms.

**Fix** (keeps the project on Turbopack): don't let the bundler see it. Vendor
the three runtime files into `public/libraw/` and spawn the worker from a **plain
string URL** (`new Worker('/libraw/worker.js')`) — Turbopack only traces the
`new URL(..., import.meta.url)` form, so a string path is treated as a runtime
asset. The worker resolves its wasm via `new URL('libraw.wasm', import.meta.url)`
relative to its own location, so the three files must stay colocated (they are).
Do **not** revert `raw.ts` to `import('libraw-wasm')` or the build hangs again.

### Verified (Playwright + pixels)
Real 30 MB Sony ARW (`ILME-FX30`, `.playwright-mcp/example-sony.ARW`):
- Decodes to **6240×4168**; preview GL canvas **2048×1368** (aspect preserved).
- Center pixel a real value `[106,79,47]`, luma variance 233 (not uniform/black).
- EXIF mapped from LibRaw: *Sony ILME-FX30 · E 18-135mm F3.5-5.6 OSS · 29mm ·
  f/5 · 1/125s · ISO 800*.
- Exposure +3 clips the center pixel to white (correct ~8× response) → flows
  through the pipeline.
- Export at Original = **6240×4168**, valid **8.1 MB JPEG** on disk (`sips`).
- No console errors/warnings; `next build` clean; `tsc` clean.

### RAW — deferred / future
- `rawImageData()` (16-bit mosaic) is available but unused — we take the
  demosaiced 8-bit path since the pipeline is 8-bit. A 16-bit pipeline is a
  bigger project (float textures) and out of scope.
- White balance uses the camera's recorded WB (`useCameraWb`). A future "as shot
  / auto / custom" WB control could expose LibRaw's WB settings.
- Extension allowlist in `raw.ts`/`Uploader.tsx` covers common formats; add more
  as needed (both places must agree).

### Security notes (from the Supabase skill)
- Only the **publishable/anon** key is in `NEXT_PUBLIC_*` (safe for the browser).
  **Never** put the service-role/secret key in a `NEXT_PUBLIC_*` var.
- `proxy.ts` calls `getClaims()` (verifies + refreshes) with nothing between
  client creation and the call — required by Supabase's SSR guidance.
- When cloud storage lands (below), every table in `public` needs **RLS** with an
  ownership predicate (`TO authenticated USING (auth.uid() = user_id)`), and
  UPDATE policies need both `USING` and `WITH CHECK`.
